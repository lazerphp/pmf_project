# План аккуратного переписывания проекта

## Короткий вывод

Переписать проект заново имеет смысл, но не как `big bang rewrite`.

У тебя уже есть рабочий эталон:

- физическое ядро без GUI: `SimulationCore`
- headless-запуск для автоматических прогонов
- JSON-конфиг сцены и поля
- метрика качества `t80` / `quality`
- пайплайн оптимизации через Python

Это очень хорошая ситуация для переписывания: можно не гадать, "как должно быть", а сверять новую реализацию со старой.

## Что важно сохранить из текущей версии

Ниже не обязательно "хорошая архитектура навсегда", но это уже полезные опорные решения:

- разделение на ядро симуляции и GUI
- headless-режим как отдельный способ запуска
- `RunConfig` и JSON как внешний контракт
- фиксированный `seed` для воспроизводимости
- `SimulationStats` и вычисление `quality`
- возможность прогонять один и тот же сценарий и визуально, и автоматически

Если переписывать, то лучше сохранять именно эти контракты, даже если внутренности будут совсем другими.

## Главный принцип

Старый проект должен стать не "мусором, который мешает", а **эталоном поведения**.

Новая версия должна появляться рядом и проходить через сравнение:

- одинаковый `RunConfig`
- одинаковый `seed`
- одинаковое число шагов или одинаковое `maxSimulationTime`
- сравнение метрик
- при необходимости сравнение траекторий на нескольких контрольных шагах

Тогда ты не потеряешь удачные нюансы, даже если уже не помнишь, откуда они взялись.

## Самая безопасная стратегия

### Этап 0. Зафиксировать текущую систему как baseline

Нужно собрать маленький набор эталонных сценариев:

- `simple`: мало частиц, простая геометрия
- `medium`: текущий основной сценарий из `examples/run_config.json`
- `stress`: больше частиц или более "жесткое" поле
- `failure-prone`: сценарий, где раньше были странности или нестабильность

Для каждого сценария стоит сохранить:

- JSON-конфиг
- `seed`
- результат headless-прогона
- ключевые метрики: `t80`, `uniqueTargetHits`, `missingTargetFraction`, `stepsExecuted`

Это будет твой набор регрессии.

### Этап 1. Описать систему не по файлам, а по ответственности

Перед переписыванием полезно разложить проект на смысловые блоки:

1. геометрия сцены
2. частица и ее состояние
3. внешние силы
4. межчастичное взаимодействие
5. коллизии со стенками и препятствиями
6. логика `spawn` / `target`
7. статистика и метрики
8. headless-runner
9. GUI-оболочка
10. JSON I/O

Если модуль нельзя кратко описать одной ответственностью, это кандидат на разрезание в новой версии.

### Этап 2. Переписывать сначала только ядро без GUI

Не надо начинать с окна, камеры и отрисовки.

Лучший порядок:

1. `Vector2`, `Rect`, базовая математика
2. `Particle`
3. геометрия сцены
4. коллизии
5. силы
6. `SimulationCore`
7. `SimulationStats`
8. `HeadlessSimulationRunner`
9. JSON-слой
10. GUI

Причина простая: headless-ядро легче проверять, оно быстрее запускается и не прячет ошибки за визуальным "ну вроде движется".

### Этап 3. Делать совместимость по поведению, а не по строчкам

Не обязательно повторять старую структуру классов 1 в 1.

Важно, чтобы новая версия сохраняла:

- допустимые входные данные
- физический смысл шагов
- базовые метрики качества
- воспроизводимость

Если новая архитектура проще, а результаты совпадают достаточно близко, это успех.

### Этап 4. Вводить сравнения как можно раньше

После каждого крупного блока полезно проверять не "компилируется ли", а "совпадает ли поведение".

Минимальные проверки:

- частица обновляет позицию и скорость предсказуемо
- зона корректно определяет `contains` / `intersects`
- коллизии не выпускают частицу за допустимую область
- статистика правильно считает `t80`
- headless-режим дает стабильный результат при фиксированном `seed`

### Этап 5. GUI переносить последним

Сейчас GUI уже отделен от ядра, и это подарок.

Перенос GUI в самом конце дает плюсы:

- физика уже проверена без окна
- меньше шума при отладке
- можно временно подключить новый core к старому GUI

Это, скорее всего, самый удобный переходный вариант.

## Практичный план по неделям или итерациям

### Итерация 1. Сохранить знания из старой версии

Результат итерации:

- список модулей и их ролей
- 3-5 эталонных конфигов
- зафиксированные headless-результаты
- список "важных странностей", которые нельзя потерять

### Итерация 2. Собрать новый минимальный core

Результат итерации:

- новые базовые типы
- новый `Particle`
- новый цикл шага симуляции без сложной физики
- простые тестовые сценарии

### Итерация 3. Перенести геометрию и коллизии

Результат итерации:

- новый коридор
- препятствия
- `spawn` / `target`
- совпадение с baseline на простых кейсах

### Итерация 4. Перенести силы

Результат итерации:

- внешнее поле
- попарные силы
- проверка стабильности на тех же `seed`

### Итерация 5. Перенести статистику и headless

Результат итерации:

- новый `HeadlessSimulationRunner`
- совпадающие метрики на эталонных сценариях
- сравнение старого и нового результата в полуавтоматическом виде

### Итерация 6. Подключить GUI

Результат итерации:

- старая или новая визуализация поверх нового ядра
- ручная визуальная проверка нескольких сценариев

### Итерация 7. Подключить оптимизатор

Результат итерации:

- Python-скрипт работает уже с новой версией
- формат результата не ломается

## Что лучше не делать

- не начинать переписывание без baseline-сценариев
- не менять одновременно физику, формат JSON и GUI
- не переносить все файлы механически
- не пытаться сразу "сделать идеально"
- не удалять старую реализацию, пока новая не умеет проходить эталонные кейсы

## Как понять, что переписывание идет правильно

Вот хорошие признаки:

- новая версия меньше зависит от случайного поведения
- каждый модуль можно объяснить в 2-3 предложениях
- новый код легче запускать в headless-проверках
- поведение на фиксированных сценариях не ломается без причины
- GUI перестает быть местом, где "магически все сходится"

## Рекомендуемое решение

Если коротко: **да, переписать заново можно и, похоже, это разумная идея, но только через параллельную пересборку с baseline-сравнением**.

Самый здоровый путь:

1. зафиксировать текущее поведение
2. собрать новый headless-core
3. добиться сопоставимых результатов
4. только потом переносить GUI и оптимизацию

Если захочешь, следующим шагом можно прямо в репозитории составить:

- карту модулей "что за что отвечает"
- список baseline-сценариев
- пошаговый backlog переписывания по файлам

Это уже можно сделать совсем предметно по текущему коду.
