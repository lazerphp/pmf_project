# Проект: Прохождение частиц через изогнутую трубу

Цель проекта: разработать модель движения частиц через S-образный коридор и подобрать параметры внешнего поля так, чтобы все частицы переходили из одного конца в другой за минимальное время.

## Текущее состояние проекта
Сейчас реализован прототип второй итерации: частицы движутся внутри S-образного коридора, взаимодействуют друг с другом через потенциал Леннарда-Джонса, ускоряются внешним потенциальным полем и отображаются в окне SFML. Геометрия коридора и зон задается в условных клетках через `Config::World`, чтобы координаты было проще читать и менять.

## Глобальная цель
Дальнейшая цель проекта: развить текущую модель в кроссплатформенное приложение на C++ с SFML и добавить автоматический подбор параметров внешнего поля так, чтобы заданная доля частиц достигала целевой области за минимальное время.

## Дорожная карта

Проект развивается по итерациям. На данный момент существуют итерации:

1. `iter-1`: идеальный 2D газ в прямоугольной плоскости.
2. `iter-2-beta`: первый прототип S-образного коридора, внешнего поля, зон и камеры.
3. `iter-2`: актуальное состояние второй итерации после рефакторинга конфигурации и перехода к сеточной геометрии.

>  **Как переключиться на версию конкретной итерации:**
> `git switch --detach iter-1`, `git switch --detach iter-2-beta` или `git switch --detach iter-2`.

## Подбор параметров и визуальная проверка

Практический workflow теперь такой:

1. Собираем основной бинарник `simulation`.
2. Запускаем Python-поиск, который много раз вызывает `simulation --headless`.
3. Скрипт сохраняет лучший `RunConfig` в JSON.
4. Этот же JSON можно отдать GUI-режиму и посмотреть поведение визуально.

Пример:

```bash
python3 scripts/random_search.py \
  --binary ./build/simulation \
  --trials 30 \
  --max-time 10 \
  --csv-output runs/random_search.csv \
  --best-config-output runs/best_run_config.json \
  --best-result-output runs/best_result.json \
  --summary-output runs/summary.json

./build/simulation --input-json runs/best_run_config.json
```

Если нужен базовый шаблон параметров, можно передать его в поиск:

```bash
python3 scripts/random_search.py \
  --binary ./build/simulation \
  --base-config runs/base_run_config.json \
  --trials 30 \
  --best-config-output runs/best_run_config.json
```

`--base-config` использует тот же JSON-формат, что и сам бинарник через `--input-json`.
`--summary-output` сохраняет короткую выжимку по лучшему прогону: quality, метрики, runtime и пути к артефактам.

### 🔹 Вариант Б: Windows (Visual Studio 2019/2022)

#### Шаг 1: Подготовка SFML
1. Скачайте версию SFML 3, соответствующую вашей версии Visual Studio (MSVC 16.0 для VS2019, 17.0 для VS2022) и разрядности (x64).
2. Распакуйте в путь без пробелов, например: `C:\Libs\SFML`.

#### Шаг 2: Открытие в Visual Studio
1. Запустите VS -> **"Открыть локальную папку"** (Open a local folder).
2. Выберите корень проекта (где лежит `CMakeLists.txt`).
3. VS автоматически настроит CMake.

#### Шаг 3: Настройка путей (если CMake не нашел SFML)
1. Меню **Проект** -> **Настроить параметры CMake** (Configure CMake Settings).
2. В открывшемся `CMakeSettings.json` найдите секцию `"configurations"` и добавьте строку:
   ```json
   "cmakeCommandArgs": "-DSFML_DIR=C:/Libs/SFML/lib/cmake/SFML"
   ```
   *(Замените путь на ваш реальный путь к папке SFML)*.
3. Сохраните файл.

#### Шаг 4: DLL и запуск
1. После сборки найдите `.exe` в папке `out/build/x64-Debug/`.
2. Скопируйте все `.dll` файлы из `C:\Libs\SFML\bin` в эту же папку (`out/build/x64-Debug/`).
   - Нужны DLL из SFML 3, например: `sfml-graphics-3.dll`, `sfml-window-3.dll`, `sfml-system-3.dll`.
3. Запустите `simulation.exe` через VS (правой кнопкой -> Запуск) или напрямую.

**Управление в программе:**
- Изменение размера окна: Пропорции сохраняются автоматически (черные поля).
- Перезапуск частиц: Клавиша `SPACE`.
- Масштабирование: колесо мыши, `+`, `-`.
- Перемещение камеры: зажать левую кнопку мыши и двигать мышь.
- Сброс камеры: `0`.
- Выход: Закрытие окна.

## Разбор текущих файлов

Ниже собрана краткая техническая карта проекта: что находится в каждом основном файле и за какую часть логики он отвечает.

### Корневые файлы

- `CMakeLists.txt`
  Описывает сборку проекта через CMake: собирает SFML-free библиотеку `simulation_core`, а GUI-часть подключает опционально через `BUILD_GUI`.

- `main.cpp`
  Точка входа в приложение. Разбирает аргументы командной строки, запускает headless-режим и, если GUI собран, может запускать окно `Simulation`.

- `README.md`
  Документация проекта: цель, статус итерации, запуск, реализованные части и план дальнейшей работы.

- `scripts/random_search.py`
  Простая Python-обвязка для batch-поиска. Запускает `simulation --headless`, использует `quality` из JSON-результата, умеет сохранять CSV, лучший `RunConfig` и лучший JSON-результат прогона.

### Заголовочные файлы (`include/`)

- `include/Config.h`
  Хранит физические и runtime-параметры: параметры сетки, коэффициенты внешнего поля, константы потенциала Леннарда-Джонса и базовые настройки симуляции.

- `include/VisualConfig.h`
  Хранит только GUI-параметры: размеры окна, ограничения камеры и цвета отрисовки SFML.

- `include/Vector2.h`
  Объявляет простой двумерный вектор `Vector2` с базовыми арифметическими операциями. Это основной тип для координат, скоростей, ускорений и сил.

- `include/Particle.h`
  Объявляет физическую частицу. Частица хранит только позицию, скорость и ускорение, без SFML-зависимостей.

- `include/Corridor.h`
  Объявляет геометрию коридора. Коридор представлен как многоугольный контур и набор сегментов стен. Предоставляет методы проверки нахождения частицы внутри области и разрешения столкновений со стенами.

- `include/Zone.h`
  Описывает прямоугольные зоны сцены. Базовый `Zone` умеет проверять принадлежность точки, `SpawnZone` выдает случайную точку спавна, `TargetZone` остается чисто геометрической зоной цели.

- `include/CameraController.h`
  Объявляет контроллер камеры: зум, панорамирование, пересчет viewport при изменении размера окна и сброс вида.

- `include/PhysicsForces.h`
  Объявляет физические функции: вычисление силы внешнего поля и попарной силы между частицами.

- `include/Simulation.h`
  Объявляет графическую оболочку над ядром: окно SFML, камера, обработка событий и отрисовка.

- `include/SimulationCore.h`
  Объявляет физическое ядро симуляции. Здесь находятся частицы, коридор, зоны, статистика, генератор случайных чисел и физический шаг.

### Исходные файлы (`src/`)

- `src/Particle.cpp`
  Реализует обновление физической частицы явной схемой Эйлера.

- `src/Corridor.cpp`
  Реализует геометрию и столкновения со стенками. Содержит проверку попадания точки в многоугольник, учет отступа от стены через `margin`, откат координат при столкновении и отражение соответствующих компонент скорости.

- `src/Zone.cpp`
  Реализует поведение прямоугольных зон. `SpawnZone` выбирает случайную точку внутри области, `TargetZone` остается простой геометрией цели.

- `src/CameraController.cpp`
  Реализует работу камеры. Поддерживает сохранение пропорций сцены при изменении размеров окна, масштабирование к точке под курсором и панорамирование мышью.

- `src/PhysicsForces.cpp`
  Реализует физику внешнего поля и межчастичного взаимодействия. Внешнее поле задается как кусочная потенциальная функция по участкам S-образного коридора. Межчастичное взаимодействие считается через потенциал Леннарда-Джонса с отсечением по дальности и защитой от слишком малых расстояний.

- `src/Simulation.cpp`
  Реализует GUI-оболочку: берет данные из `SimulationCore`, обрабатывает события окна и отрисовывает коридор, зоны и частицы через SFML.

- `src/SimulationCore.cpp`
  Реализует физическое ядро: создание геометрии, инициализацию частиц, накопление сил, шаг интегрирования, столкновения и обновление статистики попаданий.

- сохранять результаты прогонов в `csv` или `json`
- строить графики и сравнение наборов коэффициентов

Практический смысл этого этапа: Python удобнее для массовых запусков, сбора статистики, визуализации и анализа.

### Этап 4. Реализовать численный подбор параметров

Цель этапа: подобрать коэффициенты внешнего поля автоматически.

Нужно сделать:

- определить вектор параметров `theta`
- задать функцию качества `J(theta)`
- оценивать градиент по конечным разностям
- обновлять параметры правилом градиентного спуска
- сравнить результат с `random search` и `coordinate descent`

Практический смысл этого этапа: от ручного тюнинга перейти к систематическому поиску удачных коэффициентов.

## Предлагаемый технический план рефакторинга

Ниже не обязательная итоговая архитектура, а удобный практический путь без полной переписки проекта.

### 1. Выделить физическое ядро

Создать отдельный класс, например `SimulationCore`, который будет отвечать только за физику и состояние симуляции.

В него стоит перенести:

- список частиц
- коридор
- зоны
- параметры поля
- генератор случайных чисел
- инициализацию частиц
- вычисление сил
- физический шаг
- обработку столкновений
- обновление статистики достижения цели

Важно: в этом классе не должно быть `sf::RenderWindow`, обработки мыши, клавиатуры и камеры.

### 2. Оставить текущий `Simulation` как оболочку над ядром

После выделения ядра класс `Simulation` можно превратить в GUI-слой.

Он будет отвечать только за:

- создание окна
- цикл событий
- управление камерой
- запуск нескольких физических шагов за кадр
- отрисовку коридора, зон и частиц

Идея такая: физика должна быть одна и та же, а интерфейсов запуска может быть несколько.

### 3. Вынести статистику попаданий из `TargetZone`

Сейчас `TargetZone` хранит только простой счетчик попаданий. Для корректной метрики этого недостаточно.

Лучше завести отдельную структуру, например `SimulationStats`, где будут храниться:

- `reachedTarget[i]` для каждой частицы
- `firstHitTime[i]` для каждой частицы
- общее число уникально достигших цели частиц
- вычисленное значение `t80`
- итоговая функция качества

Тогда `TargetZone` останется геометрической сущностью, а не хранилищем истории симуляции.

### 4. Добавить headless/batch-режим

После выделения ядра нужно сделать запуск без окна.

Такой режим должен уметь:

- создать симуляцию с заданными параметрами
- прогнать ее до `maxSimulationTime`
- остановиться раньше, если `t80` уже достигнуто
- вернуть статистику в удобном виде

Это можно сделать отдельным `main_headless.cpp` или отдельным режимом запуска через аргументы командной строки.

### 5. Разделить runtime-параметры и compile-time-константы

Сейчас в `Config.h` смешаны:

- параметры модели
- параметры визуализации
- параметры окна
- коэффициенты поля

Для будущей оптимизации полезно отделить то, что должно настраиваться во время запуска, в отдельную структуру наподобие `RunConfig` или `SimulationConfig`.

В ней могут лежать:

- `particleCount`
- `dt`
- `physicsStepsPerFrame`
- `fieldParams`
- `maxSimulationTime`
- `seed`

### 6. Отделить физическую частицу от визуальной

Этот шаг уже сделан: `Particle`, `Zone` и `Corridor` больше не зависят от SFML, а отрисовка живет в `Simulation`.

Практический смысл результата:

- headless-ядро можно использовать отдельно от окна
- GUI стало проще держать как оболочку над `SimulationCore`

## Рекомендуемый практический порядок работ

Чтобы не распыляться, разумно идти так:

1. Исправить подсчет попаданий в `target`
2. Добавить хранение времени первого попадания для каждой частицы
3. Научиться считать `t80`
4. Вынести физический шаг из `run()` в отдельный API
5. Выделить `SimulationCore`
6. Перевести текущий SFML-интерфейс на использование ядра
7. Сделать headless/batch-режим
8. Добавить передачу параметров поля извне
9. Написать Python-обвязку
10. Реализовать и сравнить методы оптимизации

## Минимальный MVP-рефакторинг

Если идти самым коротким путем к автоматическим прогонам, достаточно сделать такой минимум:

- оставить `Particle`, `Corridor`, `PhysicsForces` и `CameraController` почти без изменений
- создать `SimulationCore`
- вынести туда физику и статистику
- оставить `Simulation` как графическую оболочку
- добавить отдельный headless entrypoint

После этого проект уже станет пригоден для пакетных прогонов и подбора параметров.
